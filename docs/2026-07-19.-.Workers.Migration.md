# Migrate GeoNotes from Cloudflare Pages to Workers

## Context

The app runs on Cloudflare Pages with file-routed Functions in `functions/`. Two things the codebase already wants are impossible there, and both are documented as Pages workarounds in the code itself:

- `functions/_lib/rate-limit.ts` is written against the native `AUTH_RATE_LIMITER` binding, which wrangler rejects for Pages projects. The binding is always undefined, so `enforceAuthAbuseLimit` no-ops on every call.
- `purgeExpiredDeletedAccounts` piggybacks on `functions/api/auth/email-request.ts` via `waitUntil` purely because Pages has no cron trigger.

Moving to a Worker with a static-assets binding unlocks both, and is the strategy already settled on for D1 abuse containment. The project is pre-launch with no users, so there is no migration risk to user data, but production is live on `gnotes.vshub.app` with WebAuthn scoped to that host, so the cutover still has to be reversible.

Scope is a **straight port**. No CORS, no bearer tokens, no Capacitor auth work: the API stays same-origin-relative and `src/lib/api.ts`'s `API_BASE`/`VITE_API_URL` seam is left intact so the future Android build still has its hook. **`TODO.md` is not to be touched.**

Branch `workers-migration` is already created off `main` at `35afe01`; the tag `pre-workers-migration` marks the last Pages commit.

## Approach

20 route files under `functions/api/` all funnel through one `route<Env>()` wrapper in `functions/_lib/http.ts`, typed on `EventContext`. Retyping that single wrapper to accept a Hono `Context` and adapt it into the same `{ env, request, params, waitUntil }` shape lets **all 20 handler bodies stay byte-identical**. That is the crux of keeping the diff reviewable.

### Commit 1 — pure renames, zero code change

File location no longer determines routing on Workers, so the Pages naming becomes actively misleading (`_middleware.ts` stops auto-running, `[id].ts` stops mapping to a URL). Rename first, separately, so commit 2 is readable:

- `git mv functions worker`
- `worker/_middleware.ts` → `worker/site.ts`
- `worker/api/auth/credentials/[id].ts` → `.../by-id.ts`, same for `.../sessions/[id].ts`
- `tsconfig.functions.json` → `tsconfig.worker.json` (`include: ["worker", "shared"]`), update the reference in `tsconfig.json`

`pnpm typecheck && pnpm test` must pass unchanged.

### Commit 2 — the port

**`pnpm add hono`** (only new runtime dep).

**`worker/_lib/env.ts`**: change `export interface Env` to `export type Env = {}`. Hono constrains `Bindings extends Record<string, unknown>` and an `interface` does not satisfy an index-signature constraint. Missing this produces the same error on all 20 routes. Add the `ASSETS: Fetcher` binding.

**`worker/_lib/http.ts`**: keep `json`, `error`, `HttpError`, `hasTrustedOrigin`, `requireTrustedOrigin` untouched (so `http.test.ts` still passes). Replace only `route()`:

```ts
export interface RouteContext<E extends RouteEnv> {
  env: E
  request: Request
  params: Record<string, string | undefined>
  waitUntil: (promise: Promise<unknown>) => void
}

export function route<E extends RouteEnv>(
  handler: (ctx: RouteContext<E>) => Promise<Response>,
): (c: Context<{ Bindings: E }>) => Promise<Response>
```

Inside: `requireTrustedOrigin(c.env, c.req.raw)`, then call the handler with `c.req.raw`, `c.req.param()`, and a `waitUntil` that falls back to a fire-and-forget `.catch(console.error)` when `c.executionCtx` throws (it does under `app.request()` in tests). `catch` block is unchanged. `params` typed `string | undefined` narrows fine against the existing `typeof params.id === 'string' ? params.id : ''` in both by-id handlers.

Also per **R2 below**, add `X-Content-Type-Options: nosniff` and `Referrer-Policy: no-referrer` to `json()` and `error()`.

**`worker/router.ts`** (new): a Hono app importing the 20 handlers by their existing `onRequestGet`/`Post`/`Delete` exports.

```ts
export const app = new Hono<{ Bindings: Env }>()
app.post('/api/sync', sync)
app.delete('/api/auth/sessions/:id', authSessionDelete)
// ...18 more; register static segments before :id params
app.all('/api/*', () => error(404, 'not found'))   // must precede the catch-all
app.all('*', (c) => serveSite(c.env, c.req.raw))
```

The `/api/*` guard is load-bearing: without it a mistyped API path falls through to the SPA fallback and returns `index.html` with a 200, so the client's `res.json()` throws a parse error instead of a clean `ApiError(404)`.

**`worker/site.ts`**: `_middleware.ts` becomes `serveSite(env, request)`, the assets-fallback handler. `next()` → `env.ASSETS.fetch(request)`; the manifest JSON rewrite, the `HTMLRewriter` title/apple-meta block, and `withoutStaleValidators` are all copied verbatim. Ordering flips (Worker runs first, API never reaches `serveSite`) but behaviour is identical, since API responses were never `text/html` nor `/manifest.webmanifest`.

**`worker/index.ts`** (new): `export default { fetch: app.fetch, async scheduled(_event, env, ctx) { ... } } satisfies ExportedHandler<Env>`. The `scheduled` handler `ctx.waitUntil`s `purgeExpiredDeletedAccounts` and `pruneExpiredEmailCodes`. `noUnusedParameters` is on, so `_event` must carry the underscore.

**Remove** the `purgeExpiredDeletedAccounts` piggyback from `email-request.ts` (a 7-statement D1 batch on the hottest unauthenticated endpoint, for nothing, once a cron exists). **Keep** `pruneExpiredEmailCodes` there, since it is genuinely amortised onto the request that grows the table, as well as in the cron. Update the now-false Pages comments in `account-deletion.ts`, `email-request.ts`, `rate-limit.ts`, and the `AUTH_RATE_LIMITER` JSDoc in `env.ts`. Keep the `if (!env.AUTH_RATE_LIMITER) return` guard: it is what lets unit tests run.

**`wrangler.toml`**: delete `pages_build_output_dir` (wrangler rejects it alongside `main`). Add `main = "worker/index.ts"`, an `[assets]` block (`directory = "./dist"`, `binding = "ASSETS"`, `not_found_handling = "single-page-application"`), `[[ratelimits]]` (`name = "AUTH_RATE_LIMITER"`, arbitrary unique `namespace_id`, `simple = { limit = 20, period = 60 }` — **`period` only validates as 10 or 60**, confirmed in wrangler's config schema), and `[triggers] crons`. `[vars]` and the D1 block are unchanged.

**Name the Worker `geonotes-worker`, not `geonotes`.** This avoids a namespace collision with the Pages project and, more importantly, lets Pages stay fully deployed in parallel, which is what makes the rollback instant.

**`package.json`**: `preview` → `wrangler dev --port 8788 --ip 0.0.0.0`, add `deploy: "pnpm build && wrangler deploy"`. `dev`, `build`, `test`, `typecheck` unchanged. **`vite.config.ts` needs no changes** — the `/api` → `:8788` proxy works identically against `wrangler dev`.

Keep the two-process vite + wrangler setup; **do not adopt `@cloudflare/vite-plugin`**. It would collapse the two servers, invalidating both the AGENTS.md contract that `dist` is built-not-watched and the nginx→8788 staging path, in the same change as a production cutover. Revisit later as an isolated DX change.

Run `pnpm wrangler deploy --dry-run` as soon as this commit compiles (see R4).

### Commit 3 — tests

`worker/router.test.ts`, plain vitest driving `app.request(path, init, env, ctx)` with a hand-rolled env. **Do not add `@cloudflare/vitest-pool-workers`** — it would drag in a second runtime and a D1 harness for four fast files, and the only thing it would additionally cover (`HTMLRewriter`, absent in node) is a cosmetic staging-only label you verify by looking at the home screen anyway.

1. `POST /api/sync` with no `Origin` → 403 `bad origin`; wrong `Origin` → 403. The CSRF chokepoint is currently only tested at the `hasTrustedOrigin` unit level, never through the router.
2. `GET /api/geocode?lat=999&lng=0` → 400, proving the shim wires `request` through without touching D1 or the network.
3. `GET /api/nope` → 404 **with `content-type: text/plain`**, not the SPA HTML. Regression guard for the fallback hazard above.
4. `DELETE /api/auth/credentials/abc123` against a fake `DB.prepare` that records bind args → asserts `params.id === 'abc123'`. Highest-value new test: path params are the one thing the shim genuinely re-implements.
5. A route called with no `ExecutionContext` does not throw (the `waitUntil` fallback).

## Verification

**Local** — `pnpm typecheck && pnpm test && pnpm build && pnpm preview`, then on `localhost:8788`: SPA loads, a deep link falls back, `/manifest.webmanifest` says "GeoNotes Dev", `/api/geocode?lat=1&lng=1` works, `/api/nope` returns text 404. Sign in with the dev email code. **Check the `wrangler dev` bindings table actually lists the rate limiter** — if miniflare does not simulate it, the `if (!env.AUTH_RATE_LIMITER) return` guard means you silently exercise the unbound path.

Local D1 survives the switch: miniflare keys its sqlite off `database_id`, which is unchanged, so `wrangler dev` opens the same file `wrangler pages dev` used.

**Staging** — restart the :8788 host process as `wrangler dev --port 8788 --ip 0.0.0.0` (the `--ip` is load-bearing; nginx reaches it via `host.docker.internal`). nginx and the tunnel need no change. On `https://geoclaude.vsantos.net`: install the PWA and confirm the "GeoNotes Dev" label on Android and iOS, then passkey register + login, sync, session revoke, deletion request. **Do not pass `--test-scheduled`** — it would expose a public `/__scheduled` that anyone could use to trigger the purge.

**Cutover** (user-run; each step reversible):

1. `wrangler secret put` AUTH_SECRET, RESEND_API_KEY, TURNSTILE_SECRET, then `secret list` to confirm. **AUTH_SECRET must be byte-identical to the Pages value** — see R3.
2. `pnpm deploy`. Creates `geonotes-worker` with **no route attached**: zero production impact, Pages still serving everyone.
3. Smoke-test the workers.dev subdomain: assets, deep links, `/api/geocode`, `/api/nope`. Passkeys and email sign-in **will** fail there (RP_ID and the `Origin` CSRF check both point at `gnotes.vshub.app`) — that is correct, do not chase it.
4. `wrangler tail` and confirm the cron fires cleanly. The purge is idempotent.
5. **Add a Worker Route `gnotes.vshub.app/*` → `geonotes-worker`.** Use a Route, not a Custom Domain: a Custom Domain cannot be attached while Pages holds the hostname, forcing a detach-then-attach with a real DNS gap. A Route sits in front of the existing Pages domain with DNS untouched, takes effect in seconds, and **rolls back by deleting the route**.
6. Verify production: manifest says "GeoNotes" (not Dev), an existing session is still signed in, passkey login and registration, email sign-in, sync, session revoke. Watch `wrangler tail` for 500s.
7. Soak a few days. Then, as a separate low-stakes change, swap the Route for a proper Custom Domain (this is the step that needs the Pages detach) and delete the Pages project.

D1 needs **nothing**: the Worker binds the same `database_id`, so it is literally the same rows.

## Risks

- **R1 — `interface Env`.** Fails typecheck on all 20 routes, easy to misdiagnose as "Hono is broken". Fix is the one-word `interface`→`type` above.
- **R2 — `_headers` scope narrows (real regression).** On Pages, `_headers` applied to Functions output too. On Workers it applies **only to asset responses**, so `/api/*` loses `nosniff`, `Referrer-Policy`, HSTS, `X-Frame-Options`. HSTS and CSP are moot on JSON, but losing `nosniff` on API responses is a genuine downgrade. Mitigated by setting the headers in `json()` and `error()`, which every route returns through.
- **R3 — AUTH_SECRET mismatch breaks WebAuthn.** It HMAC-signs challenge tokens in `challenge.ts`. A different value fails every in-flight ceremony, and during the parallel window a ceremony started on Pages and finished on the Worker breaks. It is not readable back out of Cloudflare — **find your stored copy now, not at cutover**. Sessions are unaffected either way (opaque D1 tokens, not signed).
- **R4 — `nodejs_compat` may be needed.** Pages' Functions builder applies different default polyfills than `wrangler deploy`. Run `pnpm wrangler deploy --dry-run` right after commit 2 rather than discovering this at deploy time; add the flag if a `node:` builtin fails to resolve.
- **R5 — the rate-limit binding is per-colo and eventually consistent.** An attacker spread across colos gets `limit × colos`. It is a cheap inner layer that fails fast before D1, **not** a replacement for the zone WAF rule. Do not delete the WAF rule as now-redundant.
- **R6 — `caches.default` is a no-op off-zone.** `geocode.ts` will miss on every workers.dev request. Only verifiable at step 6, not step 3.
- **R7 — service worker.** Because the migration touches no frontend source, `pnpm build` yields byte-identical content-hashed filenames, so every installed SW's precache stays valid across the cutover. **Deploy the cutover with an unmodified frontend**; do not bundle a UI change into the same deploy. `navigateFallbackDenylist: [/^\/api\//]` stays and composes with the `/api/*` 404 guard.
- **R8 — `pages.dev` is not a read-only shadow.** During the soak the Pages deployment still reaches production D1 if anyone hits `geonotes-49a.pages.dev`. Same code, same DB, so not a correctness problem, just not a sandbox.
